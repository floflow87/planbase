Je souhaite refondre le module Whiteboard de mon application pour en faire un vrai espace de travail visuel fluide, libre et exploitable, proche dans l’expérience de Miro, tout en restant connecté à mes objets métier existants.

L’objectif est de transformer le whiteboard actuel, qui ressemble encore trop à une simple vue de cartes, en un canvas libre permettant de réfléchir, structurer, organiser et ouvrir des entités métier depuis le board.

Merci de travailler en plusieurs étapes propres, robustes et maintenables, avec une attention particulière portée à la fluidité UX, à la qualité du code et à la stabilité.

1. Objectif produit

Je veux que le Whiteboard permette de :

créer et manipuler des objets libres sur un canvas infini
ajouter du texte libre
ajouter des formes libres
déplacer, redimensionner, sélectionner et organiser les éléments
connecter des éléments entre eux
représenter des objets métier sous forme de cartes compactes :
note
fiche client
tâche
projet
ouvrir ces objets métier en pop-up / drawer / full window
avoir une sensation proche d’un outil de co-construction visuelle type Miro

Le whiteboard ne doit plus être seulement une vue de données, mais un espace de travail visuel libre.

2. Problèmes actuels à corriger

Merci de corriger les problèmes existants du module actuel :

Bugs
le texte libre provoque actuellement une erreur 400
l’expérience générale n’est pas assez fluide
les objets ressemblent trop à des cartes rigides
le comportement du board n’est pas assez libre ni naturel
Limites UX actuelles
manque de liberté dans la construction
pas assez d’objets natifs de whiteboard

sensation plus “mindmap statique” que “canvas libre”
hiérarchie visuelle insuffisante

8. Architecture technique attendue

Merci de refactorer proprement le module si nécessaire.

Je veux une structure claire avec par exemple :

un modèle générique WhiteboardItem
un champ type
une structure data propre selon le type
un stockage cohérent des coordonnées et dimensions
une séparation claire entre rendu canvas et logique métier

Exemple de structure attendue :

type WhiteboardItemType =
  | "text"
  | "sticky_note"
  | "shape"
  | "connector"
  | "entity_card";

interface WhiteboardItem {
  id: string;
  boardId: string;
  type: WhiteboardItemType;
  x: number;
  y: number;
  width?: number;
  height?: number;
  rotation?: number;
  zIndex: number;
  data: Record<string, any>;
  createdAt: string;
  updatedAt: string;
}

Et pour les objets métier, un entity_card avec par exemple :

data: {
  entityType: "task" | "project" | "client" | "note",
  entityId: string,
  viewMode: "compact"
}

Merci de proposer si besoin une structure plus robuste, tant qu’elle reste simple, maintenable et évolutive.
