Tu es développeur senior fullstack sur PlanBase.

Contexte :
PlanBase est une web app SaaS pour freelances, CPO, product managers et indépendants.
L’application contient déjà des modules comme Dashboard, Projets, Backlog, Roadmap, CRM, Trésorerie.
Un premier module Feedback a été prévu, mais il doit maintenant être rattaché à la partie Backlog, et non à la partie Projet.

Objectif :
Créer une V2 du module Feedback dans la partie Backlog.
Cette V2 doit permettre de centraliser les retours utilisateurs, clients ou parties prenantes, puis de les qualifier, les regrouper et les transformer en tickets backlog.

Le module doit devenir une vraie “Feedback Inbox” orientée discovery produit.

Emplacement :
Ajouter une entrée ou un onglet “Feedbacks” dans la partie Backlog.

Structure recommandée dans Backlog :
- Tickets
- Sprints
- Feedbacks
- Insights, si déjà pertinent ou prévu plus tard

Ne pas rattacher le module uniquement à un projet.
Le feedback peut être lié :
- à un backlog ;
- à un ticket existant ;
- à un futur ticket ;
- à un projet si la relation existe, mais ce n’est pas le rattachement principal.

Nom du module :
Feedbacks

Sous-titre :
“Centralisez les retours terrain et transformez-les en tickets priorisables.”

---

## 1. Vue principale : Feedback Inbox

Créer une page “Feedbacks” dans Backlog.

Cette page doit afficher tous les feedbacks reçus ou créés manuellement.

Objectif de la vue :
Permettre à l’utilisateur de voir rapidement :
- les nouveaux feedbacks à traiter ;
- les feedbacks à qualifier ;
- les feedbacks déjà liés à des tickets ;
- les feedbacks rejetés ou archivés ;
- les demandes récurrentes.

Filtres simples :
- statut ;
- type ;
- importance ;
- source ;
- backlog lié ;
- ticket lié ;
- période ;
- recherche texte.

Vue recommandée :
Un tableau ou une liste de cards.

Colonnes principales :
- titre ;
- type ;
- importance ;
- source ;
- contributeur ;
- backlog lié ;
- ticket lié ;
- statut ;
- date ;
- actions.

## 2. Statuts des feedbacks

Ajouter les statuts suivants :

- new = Nouveau
- to_qualify = À qualifier
- qualified = Qualifié
- linked_to_ticket = Lié à un ticket
- rejected = Rejeté
- archived = Archivé

Règles :
- Tout nouveau feedback externe arrive en statut “new”.
- Un feedback peut être passé en “to_qualify” si l’utilisateur commence à le traiter.
- Un feedback devient “qualified” après analyse.
- Un feedback devient “linked_to_ticket” lorsqu’il est lié à un ticket backlog.
- Un feedback rejeté reste consultable.
- Un feedback archivé ne doit plus apparaître par défaut dans la vue principale.

- urgency : low | medium | high
- segment : freelance | agency | client | internal | other
- tags
- product_area
- backlog_id nullable
- linked_ticket_id nullable
- qualification_notes

Objectif :
Transformer un retour brut en information exploitable pour prioriser le backlog.

## 6. Créer un ticket depuis un feedback

Ajouter une action principale :

“Créer un ticket depuis ce feedback”

Au clic :
Ouvrir un drawer ou formulaire de création de ticket prérempli.

Champs préremplis :

Titre :
reprendre le titre du feedback.

Type :
- bug si feedback type bug ;
- feature ou improvement si feedback type improvement/idea ;
- task ou discovery si question/pain_point selon les types existants dans PlanBase.

Description préremplie :

Format recommandé :

En tant que [segment si connu],
je souhaite [besoin reformulé],
afin de [bénéfice attendu].

Contexte :
[description originale du feedback]

Source :
Feedback reçu de [contributeur] le [date]

Qualification :
- Impact utilisateur :
- Fréquence :
- Urgence :
- Importance :

Critères d’acceptation :
- À compléter

Règle :
Après création du ticket :
- lier automatiquement le feedback au ticket créé ;
- passer le feedback en statut linked_to_ticket ;
- afficher le lien vers le ticket.

## 7. Lier un feedback à un ticket existant

Ajouter une action :

“Lier à un ticket existant”

L’utilisateur doit pouvoir rechercher un ticket existant dans le backlog.

Recherche par :
- titre ;
- identifiant ticket ;
- backlog ;
- statut.

Après sélection :
- créer la relation feedback ↔ ticket ;
- passer le feedback en statut linked_to_ticket ;
- afficher le ticket lié dans le drawer.

Un ticket peut avoir plusieurs feedbacks liés.
Un feedback peut être lié à un seul ticket en V2 simple.
Prévoir une structure qui permettra éventuellement plusieurs relations plus tard.

## 9. Score simple de feedback

Ajouter un score simple pour aider à prioriser.

Score indicatif :
feedback_score = impact_user + frequency + urgency + importance

Mapping simple :
- low = 1
- medium = 2
- high = 3
- critical = 4 pour importance uniquement
- isolated = 1
- recurring = 3

Le score doit rester indicatif.
Ne pas créer un moteur RICE complet en V2.

Afficher :
- score faible ;
- score moyen ;
- score fort.

Libellés :
- Signal faible
- Signal moyen
- Signal fort

## 14. API backend

Créer ou adapter les endpoints internes authentifiés.

GET /api/backlog/feedbacks
Retourne les feedbacks avec filtres.

Query params possibles :
- status
- type
- importance
- backlog_id
- linked_ticket_id
- search
- period_start
- period_end

POST /api/backlog/feedbacks
Crée un feedback interne.

GET /api/backlog/feedbacks/:feedbackId
Retourne le détail d’un feedback.

PATCH /api/backlog/feedbacks/:feedbackId
Met à jour la qualification, le statut ou les notes.

POST /api/backlog/feedbacks/:feedbackId/create-ticket
Crée un ticket backlog depuis le feedback.

POST /api/backlog/feedbacks/:feedbackId/link-ticket
Lie le feedback à un ticket existant.

GET /api/backlog/tickets/:ticketId/feedbacks
Retourne les feedbacks liés à un ticket.

GET /api/backlog/feedbacks/insights
Retourne les insights feedbacks.

Sécurité :
- tous les endpoints internes doivent être authentifiés ;
- toutes les requêtes doivent filtrer par organization_id ;
- ne jamais exposer les feedbacks d’une autre organisation.
